# 视频插件全链路优化与自适应转码实施计划 (V2.0)

本项目旨在通过架构级优化，解决 VPS 环境下扫描视频导致 CPU 100% 的痛点，并构建一套基于编码感知的自动化媒体中心系统。

## 一、 核心架构目标
1. **轻量化扫描**：扫描过程与繁重的 FFmpeg 探测解耦，确保 VPS 在扫描几千部影片时 CPU 保持健康。
2. **三级自适应转码**：根据视频编码标签，智能选择「透传」、「转封装」或「重编码」，实现性能与兼容性的动态平衡。
3. **闲时算力利用**：建立异步探测队列，利用 VPS 闲暇时间自动补全媒体元数据。

---

## 二、 详细实施路径

### 阶段 1：扫描性能与流程去耦
*   **扫描逻辑改造 (MediaScanService)**
    *   引入 `p-limit` 限制目录递归并发（建议 5-10 路）。
    *   **常规扫描模式**：默认仅获取文件名、后缀及 NFO 信息，彻底跳过 `ffprobe` 调用。
    *   **增量校验**：对比数据库，若文件路径/大小未变则直接跳过，实现秒级二次扫描。
*   **后台异步探测系统 (ScanQueueService)**
    *   新增后台队列：扫描结束后，将 `v_codec` 为空的条目加入队列。
    *   **并发控制**：默认 1-3 路并发（用户可调），并设 `PROBE_DELAY` (如 500ms) 缓解 CPU 突发压力。

### 阶段 2：三级智能转码路由
*   **后端决策升级 (TranscodeService)    - **三级智能分流算法 (客户端能力感知型)**：
        1. **透传 (Direct/Proxy)**: 
           - 情况 A: `v_codec == 'h264' && container == 'mp4'`
           - 情况 B: **客户端报告支持 H.265 硬件解码** (如 Safari/高级桌面浏览器) 且资源为 H.265。
        2. **转封装 (Transmux)**: `v_codec == 'h264' && container == 'mkv'` -> 仅更换容器，CPU 极低（Copy 模式）。
        3. **全转码 (Transcode)**: 资源编码不支持，且客户端无法硬解时 -> 执行重编码。确保在不支持 H.265 的设备上正常播放。
*   **前端逻辑重构 (NetdiskPlayer)**
    *   **标签驱动**：播放前请求后端「最佳播放链路」，后端根据数据库标签（v_codec）返回 HLS 或 Direct URL。
    *   **起播探测与存盘**：若该片从未被探测，起播时瞬时触发探测并存入数据库，实现「一次播放，永久标记」。

### 阶段 3：图片采集鲁棒性与纠偏
*   **ImageCacheService 优化**
    *   **原子性写入**：采用「先写临时文件再重命名」策略，彻底杜绝缓存图片损坏导致的显示异常。
    *   **自我修复**：读取缓存时若 Sharp 解析失败，自动执行删除并重下。
*   **元数据交互增强**
    *   **单片编辑接口**：支持手动修改标题、海报 URL 和简介。
    *   **锁定机制 (Lock Tag)**：手动修改过的条目自动加锁，后续全量扫描时不再覆盖用户修改的内容。
    *   **在线匹配纠偏**：详情页集成 TMDB 备选列表，一键修正识别错误的资源。

---

## 三、 技术风险对策
*   **I/O 冲突**：异步探测与前端播放可能同时竞争文件句柄。通过 `p-limit` 和 `axios` 统一调度来控制总连接数。
*   **Token 失效率**：WebDAV 长期挂起探测可能遇到 401。需在探测任务中增加 Token 自动预判与刷新。
*   **进程残留**：建立 `AbortController` 钩子，在扫描任务或转码会话关闭时强行回收 FFmpeg 僵尸进程。
